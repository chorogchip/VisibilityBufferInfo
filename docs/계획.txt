+ 나중에 실험을 할 때는 씬에서 위치를 잡는데 많은 신경을 쓰자. 씬을 렌더링하면서 실제로 탐색, 카메라 위치와 각도를 찾아야 한다.
+ 실제 권위 있는 렌더러(ground turso)를 돌려서 프레임 타임을 측정하는 것도 의미가 있겠다.
+ 실험의 재현성을 최대한 끌어올리자. 프로그램이 영속성을 유지해서 나중에 인자만 넣는 것으로도 100% 재현 가능하게 만들자.
+ 입력 한 번으로 그래프까지 나오는 end-to-end 파이프라인을 구축하자. 그래서 그 입력만 들고 있으면 복원 가능하도록.
+ 각 주체들(DOOM, ...) 에서 쓴 포맷이나 방법들을 같이 소개하자
+ 3주차 발표는 어느 정도 완결성을 띄면서 (만약 새롭게ㅣ VB기법을 도입하는 걸 고려할 경우 판단을 어떻게 해야하나?) 에 도움이 되려고 하는 발표를 하자. 기본 원리와 출발점이 뭐고, 다른 엔진은 뭐를 썼고, 실험은 어떻게 했고 데이터는 어떻고 결론이 뭐고 이렇게 완결성을 만들자.

1. 파이프라인 개선

- plotting 개선 (2순위)
- 파이프라인 더 자동화 (2순위)

2. 비교 렌더러 어떻게 할 것인가
- 서치해서 비교 렌더러 구하기 (3순위)

3. 씬 어떻게할것인가?
- attribute 측정해야 (2순위)
- 돌아다니면서 overload point 측정해야 (2순위)
- scene을 더 모으기 (3순위)
- 실험에 맞는 synthesized scene 생성 (2순위)

4. 비교 뭐뭐 할것인가?
- 현재 있는 6개 variant로 가고, 추후 VB를 더 확장하는 방식으로 가자
- lighting에 대한 구현은 나중 확장으로 빼고, lighting 부하 없음으로 인한 비현실성은 ground turso 렌더러로 증명해 보완하자 (5순위)
	즉 실제 씬에 실제 라이트를 써서 권위 있는 렌더러를 사용해 수치를 뽑자
	실험값/부하값 잘 세팅하고 잘 비교하고 잘 설명하는 것으로 커버 가능하지 않을까 한다.
- alpha blending은 독립적으로 판단, 제일 후순위 (5순위)

5. VB알고리즘 개선
- visbuf 포맷: ID / ID+bary / derivative 등등의 접근 있다
- mesh 포맷: simple, buffer추가, GPU-driven, meshlet/cluster 등의 접근 있다
- barycentric : 마찬가지로 여러 접근
- material binning : uber shader, global list, binning, tiled packing, depth buffer 등

이 네 가지가 제일 중요하다고 생각한다. 피드백에서 부족하다고 지적받았던 게 이런 면인 것 같다
하지만 이걸 다 구현하고 실험할 수는 없고 점진적으로 하나씩 추가하면서 실험해야한다
즉 현재 실험 중에서 할 수 있는 걸 하고 그 다음으로 넘어가야 한다 (3순위)

6. 현재 상황에서 개선할 것
- 1순위는 아무래도 barycentric 계산인 것 같다. 지금 되어있는 게 절대적인 구현도 아니고 아무래도 노이즈가 있다.
- material 관련 개선도 1순위다.
- draw call batching / instancing 관련도 1순위다.

이걸 아주 절대적인 무언가로 만들려고 하면 끝도 없고 불가능하다.
따라서
1. ground turso 로 쓸 수 있는 권위적인 렌더러를 몇 개 도입해서 같이 성능 테스트를 하며 비교분석한다
2. 각각의 실험에 집중해서, 실험에서 알아내려는 것에 필요한 최소한을 매번 점진적으로 구현하면서 실험해나간다
3. 불완전성은 인정한다. 단 어떤 면에서 불완전하고 현실 재현성이 부족했는지는 확실히 기록한다.
4. 적어도 어떤 면에서 부하가 얼마나 걸리고, 이게 실제 권위 있는 렌더러에선 얼마나 있었고 이런 것이 필요하다.

# VisibilityBufferInfo 웹 AI용 코드 구조 요약

이 문서는 코드 저장소에 접근할 수 없는 AI에게 VisibilityBufferInfo의 코드 구조와 각 모듈의 역할을 설명하기 위한 문서다. 실제 코드를 직접 보여주지 않아도, AI가 어떤 구조로 프로젝트가 구성되어 있고 어떤 부분을 건드려야 할지 추론할 수 있게 하는 것이 목적이다.

## 1. 전체 구조 개요

프로젝트는 Windows Direct3D 12 기반 C++20 렌더링/벤치마크 프로그램과 Python 실험 runner로 구성된다.

큰 실행 흐름:

1. Win32 application이 시작된다.
2. command-line argument를 `ProgramArgument`로 파싱한다.
3. argument validator가 값의 범위를 검사한다.
4. Win32 window를 만든다.
5. `renderer_variant`에 따라 renderer factory가 renderer를 선택한다.
6. renderer가 D3D12 device, swapchain, command queue, frame resources를 초기화한다.
7. scene을 생성하거나 외부 파일에서 로드한다.
8. CPU scene data를 GPU buffer/texture로 업로드한다.
9. render loop에서 camera update, pass별 command recording, GPU timestamp, present를 반복한다.
10. warm-up 이후 measurement frame의 timing을 모은다.
11. 종료 시 CSV와 sidecar CSV를 기록한다.
12. Python runner가 여러 parameter 조합을 반복 실행하고 결과를 보존한다.

## 2. 빌드와 외부 의존성

빌드 시스템은 CMake다.

- 프로젝트 이름: `TVBPerf`
- 언어: C++20, RC
- 플랫폼: Windows
- 주요 API: Direct3D 12, DXGI, D3DCompiler, Win32
- 외부 라이브러리:
  - Assimp: 외부 scene import
  - DirectXTex: texture decoding/loading

CMake는 `src`와 `include` 아래의 C++ 파일, `assets/shaders` 아래의 HLSL 파일을 자동 수집한다. HLSL 파일은 Visual Studio에서 빌드 도구로 컴파일하지 않고, runtime asset으로 출력 폴더에 복사된다.

빌드 결과물에는 `assets/shaders`만 복사된다. 실제 scene asset은 `assets/scenes`에 있어야 하지만 git에는 포함되지 않는 로컬 asset으로 취급된다.

## 3. 실행 인자와 결과 구조

실행 인자는 C++의 `ProgramArgument` 구조가 기준이다. Python runner는 인자 목록과 기본값을 복제하지 않고, experiment spec의 key와 runner override를 command-line으로 전달한다.

중요한 인자:

- `renderer_variant`: 사용할 renderer 선택
- `to_use_scene`: synthetic scene 대신 외부 scene을 쓸지 여부
- `to_load_texture`: 실제 texture loading 여부
- `use_vfc`: CPU view-frustum culling 사용 여부
- `scene_path`: scene 파일 경로
- `warmup_frames`: 측정 전 버릴 frame 수
- `measure_frames`: 측정할 frame 수
- `auto_terminate`: 측정 후 자동 종료
- `vsync`: VSync 여부
- `camera_mode`: free/record/playback
- `camera_filepath`: camera path CSV
- `profile_window_frames`: playback sidecar CSV의 window 크기
- `window_width`, `window_height`: 해상도
- `object_count`, `material_count`, `geometry_count`, `overdraw_count`, `geometry_div`: synthetic scene 변수
- `gbuffer_cnt`: benchmark Deferred G-buffer target 수
- `texture_count`, `texture_size`, `texture_sampling_count`, `alu_calc_count`: synthetic material workload 변수

결과 구조:

- `ProgramResult`에는 renderer 이름, 시간 통계, pass별 평균 시간이 들어간다.
- pass field는 최대 32개다. slot 0은 total time이며, slot 1부터 renderer별 pass가 기록된다.
- 종료 시 기본 result CSV를 쓴다.
- playback mode에서는 frame window별 timing sidecar CSV도 쓴다.
- scene fingerprint CSV도 output filepath에서 파생된 이름으로 저장된다.

## 4. Application과 Window

Application 계층의 역할:

- texture loader 초기화/종료
- command-line argument 파싱
- argument validation
- Win32 window 생성
- renderer factory 호출
- render loop 실행
- keyboard/mouse camera input 처리
- 자동 종료 조건 확인

Window 계층의 역할:

- Win32 window class 등록
- message pump 처리
- key state 저장
- right mouse drag 기반 camera rotation delta 기록
- ESC/close 처리

camera playback mode에서는 interactive camera input을 무시한다.

## 5. Renderer Factory와 Variant

`renderer_variant`는 renderer factory에서 concrete renderer로 변환된다.

현재 variant 의미:

- 1: Forward
- 2: Forward + Depth Pre-Pass
- 3: Deferred
- 4: TVB direct shading
- 5: Deferred + Depth Pre-Pass
- 6: TVB + G-buffer
- 7: Donut Deferred
- 8: Donut Deferred + Depth Pre-Pass
- 9: Donut Visibility + G-buffer
- 10: Benchmark Raster Stats

1~6은 benchmark/synthetic 계열 renderer다. 7~9는 실제 scene/Donut 계열 renderer다. 10은 benchmark scene을 대상으로 GPU compute에서 overdraw/quad 통계를 직접 세는 진단용 renderer다.

## 6. RendererBase

모든 renderer는 공통 base를 가진다.

RendererBase의 책임:

- D3D12 device 생성
- graphics queue 생성
- command allocator/list 생성
- swapchain 생성
- back buffer를 GPUResource로 wrapping
- viewport/scissor 설정
- camera 초기화
- camera path controller 초기화
- frame counter 초기화
- GPU frame timer 초기화
- render loop 한 frame의 공통 처리
- total timestamp 시작/종료
- present와 fence synchronization
- measurement 수집
- 종료 시 CSV 출력

RendererBase는 세 개의 virtual hook을 둔다.

- `init1_`: renderer별 resource와 pass 초기화
- `render_prepare_`: frame마다 constant buffer, culling 등 준비
- `render_record_`: 실제 pass command recording

이 구조 덕분에 각 renderer는 공통 D3D12 setup을 반복하지 않고, 자신의 pass와 resource만 구성한다.

## 7. Benchmark Renderer 계층

Benchmark renderer는 synthetic scene과 단순 benchmark용 외부 scene 경로를 담당한다.

공통 base:

- `RendererBenchmark`

RendererBenchmark의 책임:

- frame RTV/DSV/SRV/sampler manager 초기화
- depth buffer 생성
- frame constant buffer 생성
- scene loading
- scene fingerprint CSV 작성
- scene GPU resource upload
- dummy texture 또는 실제 texture 준비
- benchmark material sampler 생성
- CPU view-frustum culling 적용
- visible index count profile 갱신

Benchmark scene data:

- CPU scene은 material, vertex, index, mesh, object, batch를 가진다.
- object는 mesh와 material을 참조하고 transform을 가진다.
- batch는 같은 mesh/material 조합을 묶어 instanced draw에 사용한다.
- `all_batches`와 현재 `batches`를 분리해 culling 전/후 상태를 다룬다.
- scene은 mesh AABB와 frustum culling을 지원한다.

Benchmark renderer 종류:

- `RendererForward`: Forward와 ForwardPrepass를 옵션으로 처리
- `RendererDeferred`: Deferred와 DeferredPrepass를 옵션으로 처리
- `RendererVisBuf`: visibility + resolve
- `RendererVisBufGBuffer`: visibility + G-buffer + lighting

Benchmark pass 종류:

- `PassDepthPre`: depth-only pre-pass
- `PassForward`: forward material shading
- `PassGBuffer`: Deferred geometry/G-buffer fill
- `PassDeferredLighting`: fullscreen deferred lighting
- `PassVisibility`: visibility buffer 작성
- `PassVisBufResolve`: visibility buffer를 직접 output으로 resolve
- `PassVisBufGBuffer`: visibility buffer에서 G-buffer 생성

## 8. Donut Renderer 계층

Donut renderer는 실제 scene과 Donut-derived shader/data layout을 담당한다.

공통 base:

- `RendererDonut`

RendererDonut의 책임:

- Donut용 depth buffer 생성
- frame/shader/sampler manager 초기화
- graphics queue 외에 compute queue 준비
- Donut scene import
- Donut CPU scene validation
- Donut GPU scene resource upload
- Donut material texture descriptor heap 크기 계산
- Donut concrete renderer 초기화 호출

현재 연결된 concrete renderer:

- `RendererDonutDeferred`

RendererDonutDeferred의 책임:

- Donut Deferred 또는 Donut Deferred Prepass renderer 이름 설정
- pass name/timestamp slot 설정
- depth/G-buffer/lighting constant buffer 생성
- Donut G-buffer textures 생성
- shadow/IBL/AO/exposure fallback resource 생성
- HDR color buffer 생성
- Donut depth prepass, G-buffer, deferred lighting, tonemap pass 초기화
- frame마다 view/projection 관련 Donut constants 갱신
- depth, geometry, lighting, tonemap pass를 순서대로 recording

Donut pass 종류:

- `PassDonutDepthPre`
- `PassDonutGBuffer`
- `PassDonutDeferredLighting`
- `PassDonutTonemap`
- `PassDonutVisibility`
- `PassDonutVisGbuffer`

주의: Donut visibility 관련 pass 파일은 존재하지만, factory에 연결된 완성 renderer인지 항상 확인해야 한다. 파일이 존재한다고 실험 variant로 완성된 것은 아니다.

## 9. Scene 시스템

Scene 시스템은 benchmark scene과 Donut scene으로 나뉜다.

Benchmark scene:

- 단순 Vertex 구조: position, normal, uv, tangent
- Mesh: vertex/index range와 local AABB
- Object: mesh/material index와 transform
- ObjectBatch: 같은 mesh/material object들을 instanced draw로 묶기
- SceneBuilder: synthetic sphere/fullscreen quad scene 생성
- SceneAssimpImporter: 외부 scene을 benchmark 구조로 import
- SceneLoader: ProgramArgument에 따라 synthetic 또는 외부 scene 선택
- SceneResourceBuilder: CPU data를 GPU default heap buffer와 texture로 업로드
- SceneFingerprint: object-space scene 통계 CSV 생성

Donut scene:

- Node: scene graph tree, parent/children, local/world transform, subtree AABB
- Instance: node와 mesh 연결, world transform, geometry instance range
- Mesh: submesh range
- Submesh: global vertex/index range, material ID, local AABB
- GeometryInstance: instance와 submesh의 조합
- Material: base color, emissive, roughness, metalness, normal scale, occlusion, texture slots
- Texture: path, color space, fallback, embedded 여부
- Global vertex/index/material/texture buffers
- Draw list: geometry instance 기반 visible draw를 정렬해 GPU draw 정보로 만든다.

Donut GPU scene은 shader가 접근하기 쉬운 global buffer 형태로 변환된다. vertex data는 position, previous position, texcoord, packed normal, packed tangent stream을 가진 SoA 성격의 layout이다.

## 10. Resource와 Descriptor 관리

주요 helper:

- `GPUResource`: ID3D12Resource와 현재 resource state를 함께 저장하고 transition을 수행한다.
- `GPUQueue`: command queue, fence, wait/signal을 감싼다.
- `ResourceManagerFrame`: RTV/DSV descriptor heap 관리
- `ResourceManagerShader`: shader-visible SRV/UAV descriptor heap 관리
- `ResourceManagerSampler`: sampler descriptor heap 관리
- `ResourceViewBuilder`: buffer/texture SRV/UAV desc 생성
- `RootSignatureBuilder`: root parameter를 fluent builder 방식으로 구성
- `GraphicsPipeline`: PSO 설정과 build
- `UploadConstBuf`: upload heap constant buffer를 반영구적으로 map하여 업데이트

중요한 설계:

- descriptor 위치는 enum으로 고정된다.
- Donut descriptor 영역과 benchmark descriptor 영역이 같은 heap 안에서 구분된다.
- pass는 자신이 필요한 SRV/RTV/DSV/sampler를 생성하고 root signature를 구성한다.
- resource state 전이는 `GPUResource::transition`을 통해 추적한다.

## 11. Shader 구조

Shader 파일은 `assets/shaders` 아래에 있다.

Benchmark shader:

- forward vertex/pixel shader
- deferred geometry/lighting shader
- depth prepass shader
- visibility pass shader
- visbuf resolve shader
- visbuf G-buffer shader
- common material/input/barycentric/G-buffer include

Donut shader:

- `assets/shaders/donut` 아래 Donut-derived shader와 include
- Donut G-buffer, deferred lighting, tonemapping 관련 shader
- 일부 프로젝트용 wrapper 또는 수정 shader

C++과 HLSL 사이에서 중요한 것:

- constant buffer struct layout
- StructuredBuffer element layout
- ByteAddressBuffer offset
- register 번호와 register space
- descriptor table 순서
- root parameter enum 순서
- material texture descriptor count

## 12. Camera와 FrameCounter

Camera 시스템:

- perspective projection
- view matrix
- WASD/Space/Shift 이동
- mouse drag 회전
- camera pose 저장/복원

Camera path:

- CSV로 keyframe 저장
- record mode에서 일정 frame 간격으로 pose 기록
- playback mode에서 frame에 따라 pose interpolation
- 반복 가능한 실험에 필수

FrameCounter:

- warm-up frame은 버린다.
- measurement frame만 저장한다.
- pass별 frame time vector를 모은다.
- min/median/max/avg/percentile을 계산한다.
- playback mode에서는 windowed average CSV를 만들 수 있다.
- visible index count profile도 함께 기록할 수 있다.

## 13. Python Runner

Python runner는 `scripts/run.py` 하나가 중심이다.

역할:

- JSON experiment spec 읽기
- base/sweep/samples 해석
- spec key와 runner override를 command-line argument로 변환
- TVBPerf.exe 반복 실행
- 임시 output CSV 생성
- result CSV 읽기
- combined CSV에 누적
- run report JSON 작성
- per-run CSV와 sidecar CSV 복사
- 실패, timeout, interrupt, salvaged 상태 기록

JSON spec 구조:

- `executable`: 실행 파일 경로
- `repeat`: 반복 횟수
- `keep_individual_csv`: 개별 CSV 보존 여부
- `base`: 모든 run에 공통 적용할 argument
- `sweep`: Cartesian product로 돌릴 argument 목록
- `samples`: 명시적 sample 목록
- `timeout_seconds`: 선택적 timeout

특징:

- JSON key에서 hyphen은 underscore로 normalize된다.
- `_`로 시작하는 key는 무시된다.
- ProgramArgument 인자 이름과 값의 유효성 검증은 Python이 아니라 C++ parser/validator가 담당한다.
- unknown argument는 error다.
- `sweep`과 `samples`를 동시에 쓰면 error다.
- result row가 있더라도 stderr/log가 있으면 `salvaged`로 기록될 수 있다.

중요한 결과물:

- 실험별 combined CSV
- run report JSON
- per-run raw CSV
- playback windowed sidecar CSV
- scene fingerprint CSV
- plot PNG/SVG
- analysis report Markdown/HTML
- ML/regression 결과 CSV

결과 디렉터리 성격:

- `scripts/results/succeed`: 성공한 실험 결과

## 16. 현재 코드 관점의 연구 상태

코드 관점에서 프로젝트는 두 축으로 나뉜다.

첫 번째 축은 synthetic/benchmark 실험이다. 이 축은 이미 Forward, Deferred, TVB 계열 비교가 가능하고, 변수 sweep과 pass별 timestamp 분석에 적합하다.

두 번째 축은 실제 scene/Donut 실험이다. 이 축은 실제 PBR material과 scene hierarchy를 포함해 Deferred baseline을 만들고, 여기에 visibility buffer 계열을 붙이는 방향이다. 현재 Donut Deferred는 구성되어 있으나, Donut visibility 계열은 더 완성하고 연결해야 한다.

따라서 코드 접근 권한이 없는 AI가 이 프로젝트를 도울 때는 다음처럼 생각하면 된다.

- 실험 설계/분석: renderer variant, scene, camera path, VFC, texture loading, resolution, warm-up/measure frame을 명확히 묻는다.
- 이론 설명: Forward의 overdraw material 비용, Deferred의 G-buffer bandwidth, TVB의 indirect resolve 비용을 비교한다.
- 실제 scene 조언: scene fingerprint, screenshot/video, visibility visualization을 요구한다.
- 코드 구조 조언: renderer/pass/resource manager/scene builder/Python runner의 책임을 분리해 설명한다.

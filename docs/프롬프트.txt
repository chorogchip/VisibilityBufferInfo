# VisibilityBufferInfo 프로젝트 컨텍스트

프로젝트 저장소는 `https://github.com/chorogchip/VisibilityBufferInfo`이며, 이 문서는 2026년 7월 14일 `master`, commit `97ccf24c2eadddcfd625e7f558119cfe199f14e4` 전후의 상태를 기준으로 한다.

## 프로젝트 목적

VisibilityBufferInfo는 Triangle Visibility Buffer(TVB) 기법이 기존 Forward 및 Deferred 렌더링보다 성능상 유리하거나 불리해지는 조건을 실험적으로 규명하기 위한 프로젝트다.

핵심 비교 대상은 overdraw와 2×2 pixel quad로 낭비되는 fragment shading, depth pre-pass 이후의 두 번째 geometry 처리와 rasterization, Deferred G-buffer bandwidth, TVB resolve의 vertex·index·object·material 간접 접근, cache locality, barycentric·perspective correction·attribute derivative 재구성, material 분류와 batching 및 draw-call·PSO 관리 비용이다.

목표는 변수에 따라 유리한 알고리즘이 바뀌는 crossover point를 찾고, 측정 데이터에 맞는 성능 공식을 수립하며, VRAM bandwidth·ALU·texture sampling·rasterization·L2 cache 관점의 roofline 분석으로 결과를 설명하는 것이다. 최종적으로 알고리즘 A와 B의 성능 관계가 변수 `v`에 따라 어떻게 변하는지, 전환 지점이 어디인지, 이를 어떤 공식과 하드웨어 병목으로 설명할 수 있는지를 제시한다.

## 개념적 관점

렌더링 과정은 geometry와 visibility 처리, material attribute 계산과 texture sampling 및 shading, lighting과 output 계산의 세 단계로 본다.

Forward는 이 단계들이 결합되어 가려지는 fragment에도 material 비용이 발생할 수 있고, Deferred는 material 결과와 lighting 사이에 G-buffer를 두며, TVB는 visibility 처리와 material 계산 사이에 ID buffer를 두어 최종 visible sample을 중심으로 material 계산을 수행한다.

TVB의 기대 이점은 overdraw와 quad inefficiency로 버려지는 material 계산 감소, depth pre-pass 이후의 두 번째 rasterization 대체, 직접 shading 시 G-buffer bandwidth 감소다. 주요 비용은 geometry와 material 데이터의 indirect access, 낮은 cache locality, barycentric과 derivative 재구성, fullscreen resolve, material classification과 dispatch 관리다.

## 연구 구성

### Synthetic Primitive Scene과 Custom Renderer

독립 Direct3D 12 렌더러에서는 lighting과 복잡한 실제 material을 제외하고, sphere와 fullscreen quad를 이용해 geometry, overdraw, triangle 크기, resolution, texture sampling, ALU, G-buffer 크기, geometry working set, object·mesh·material 수를 통제한다.

Synthetic scene은 변수 사이의 영향을 분리하기 위한 장면이다. Sphere scene은 triangle subdivision, geometry working set, 공간 분포를 변화시키는 데 사용하고, quad scene은 화면을 덮는 primitive를 깊이 방향으로 배치해 overdraw와 정렬 순서를 제어하며 subdivision으로 projected triangle 크기를 변화시키는 데 사용한다.

이 장면에서는 변수 sweep을 통해 crossover point를 찾고, pass별 고정 비용과 입력 크기에 비례하는 비용을 분리해 데이터 기반 성능 모델을 만든다. 측정 결과는 VRAM bandwidth, ALU, texture sampling, rasterization, L2 cache와 연계해 roofline 형태로 분석한다.

### Intel Jungle Ruins와 Bistro, Donut Deferred Renderer

Synthetic scene은 실제 게임 렌더러의 geometry·material·texture·overdraw 분포와 lighting 비용을 그대로 재현하지 않는다. 현실적인 부하와 synthetic 실험에서 고정값으로 남아 있던 비용은 NVIDIA Donut renderer에서 측정한다.

Donut의 기존 Deferred renderer를 baseline으로 사용하고 여기에 Visibility Buffer 방식을 추가한다. Intel Jungle Ruins는 overdraw가 큰 조건을 중심으로 사용하고, Amazon Lumberyard Bistro는 보다 일반적인 장면 조건을 다루는 데 사용한다.

각 장면에서는 geometry, material, texture, triangle 크기, overdraw와 공간적 분포를 scene fingerprint로 수치화한다. Synthetic scene에서 수립한 성능 모델이 실제 렌더러와 실제 장면에서도 적용되는지 가설을 세우고 검증하며, 맞지 않는 경우 누락된 변수와 원인을 찾는다.

### 공통 분석 범위

두 연구 구성 모두 GPU hardware profiling을 동반하며, VRAM traffic, cache behavior, texture sampling, ALU, rasterization과 관련된 hardware counter를 성능 결과와 함께 해석한다.

TVB가 유리한 장면과 불리한 장면을 모두 찾으며, 예측과 결과가 일치하지 않을 경우 projected geometry, overdraw, material locality, cache layout, buffer layout, texture 특성, draw-call 구조 등의 원인을 분석한다.

후속 TVB variant로 buffer format, barycentric 및 derivative 계산 방식, direct shading과 G-buffer 생성 방식, material binning, material sorting, work queue, visibility format 구성을 비교한다.

각 실험은 실험 의도, 비교 근거, 고정 조건, 독립 변수, 측정 데이터와 해석 근거를 함께 가진다. 장면과 카메라 조건은 이미지나 캡처를 통해 시각적으로 제시하며, 측정값이 지나치게 작거나 비현실적이지 않은지, 이론적 비용과 부합하는지 함께 검사한다.

## 현재 실험 렌더러

현재 렌더러는 Windows Direct3D 12와 hardware rasterizer를 사용하며, Forward, Forward+Prepass, Deferred, Deferred+Prepass, TVB, TVB+G-buffer의 여섯 renderer variant를 제공한다.

같은 mesh와 material을 사용하는 object는 instancing으로 처리되며, 모든 주요 renderer는 동일한 synthetic texture sampling 및 ALU workload를 사용한다. 주요 workload 변수는 texture 수, texture 크기, sampling 횟수, ALU 반복 횟수다.

현재 TVB는 visibility pass와 fullscreen pixel shader resolve로 구성된다. Visibility pass는 object와 primitive를 식별할 수 있는 정보를 저장하고, resolve pass는 이를 이용해 geometry와 material 데이터를 참조한 뒤 barycentric, perspective-correct interpolation, UV derivative와 texture sampling을 재구성한다. TVB 결과는 직접 output을 계산하거나 G-buffer를 생성하는 경로로 사용할 수 있다.

현재 독립 실험은 lighting과 material binning이 없는 baseline을 중심으로 하며, 초기 실험에서는 material과 mesh를 하나로 제한한다.

## Scene 구성

Synthetic scene은 seed, object 수, mesh 수, material 수, primitive 크기, subdivision, XY·Z 분포를 조절할 수 있다. Sphere scene은 geometry 복잡도와 working set 변화에, quad scene은 overdraw·정렬 순서·triangle 크기 변화에 사용된다.

외부 scene은 Assimp importer를 통해 로드할 수 있으며, scene fingerprint에는 mesh·object·material·vertex·index·triangle 수, bounding box, triangle area 분포, mesh별·material별 triangle 분포, texture reference, unique texture 수, index와 vertex 비율 등의 object-space 통계가 포함된다.

Projected triangle size, visible triangle 수, overdraw, quad efficiency, material의 화면상 공간 분포는 카메라와 해상도에 의존하므로 screen-space 측정으로 다룬다.

## 실행 및 측정

프로그램은 argument parsing, scene 생성 또는 로드, resource 생성, warm-up, frame 측정, CSV 기록, 자동 종료 순서로 실행된다.

Python `scripts/run.py`는 JSON experiment specification의 parameter Cartesian product와 repeat를 실행하고 결과를 CSV로 누적한다. 실험 결과에는 실행 파일과 commit, build configuration, 주요 argument, seed, scene과 fingerprint, renderer variant, 해상도와 카메라, warm-up과 measurement 조건, GPU와 driver, raw CSV와 plotting 입력이 함께 보존된다.

주요 측정 대상은 renderer total GPU time, pass별 GPU time, 입력 크기로 정규화한 비용, CPU submission time, draw-call과 PSO switch 수, projected triangle size, overdraw, shaded fragment와 quad efficiency, visible triangle 수, memory traffic과 hardware counter다.

# 실험 계획

## 실험 1: 두 번째 Rasterization과 Quad Waste

목적은 Forward+Prepass의 두 번째 geometry 처리와 rasterization 비용을 TVB visibility pass와 fullscreen resolve 비용에 비교하는 것이다.

Material과 mesh를 하나로 고정하고 많은 instance를 사용한다. 화면을 덮는 quad의 수로 overdraw를 변화시키고, random·front-to-back·back-to-front 순서를 비교하며, subdivision과 resolution으로 triangle당 pixel 수를 조절한다.

Forward, Forward+Prepass, TVB의 total과 pass별 시간을 측정하고, object 수와 triangle 수에 따른 scalability, normalized cost, 고정 비용과 선형 비용, crossover point를 모델링한다.

## 실험 2: 버려지는 Fragment의 Material 비용

목적은 overdraw와 quad inefficiency로 최종 결과에 기여하지 않는 fragment의 texture sampling과 ALU 비용이 각 renderer에 미치는 영향을 측정하는 것이다.

Material과 mesh를 하나로 고정하고 실험 1에서 정한 geometry, triangle 크기, resolution, overdraw 조건을 사용한다. Texture sampling 횟수, texture 수와 크기, ALU 반복 횟수를 독립적으로 변화시킨다.

Forward, Deferred, TVB를 비교하며 material workload가 증가할 때 최종 visible sample을 중심으로 shading하는 방식의 이점과 crossover point를 분석한다.

## 실험 3: Draw Call, Batching과 Material Scalability

목적은 mesh, material, batch, draw-call과 PSO switch 수가 증가할 때 CPU와 GPU 비용이 어떻게 증가하는지 측정하는 것이다.

Object 수, mesh 수, material 수, batch 수, instancing 방식, draw-call 수, PSO 수와 material의 화면상 공간 분포를 변화시킨다.

이 실험은 material별 shader와 PSO, 비교 가능한 batching 구조와 TVB material 처리 방식이 구성된 조건에서 진행하며 CPU submission과 GPU 시간을 함께 분석한다.

## 실험 4: G-buffer Bandwidth

목적은 Deferred의 G-buffer read/write 비용과 TVB의 direct shading 또는 G-buffer 생성 비용이 resolution과 buffer 크기에 따라 어떻게 변하는지 측정하는 것이다.

Deferred, TVB direct shading, TVB 기반 G-buffer 생성 경로를 비교하고, G-buffer target 수, format, bytes per pixel, resolution을 변화시킨다.

이론적 bytes per frame과 bytes per pixel을 계산하고 L2 cache 크기, VRAM bandwidth와 실제 pass 시간을 연결해 분석한다.

## 실험 5: Indirect Access와 Geometry Working Set

목적은 TVB resolve의 vertex, index, object, mesh 데이터 간접 접근이 geometry working set과 cache locality에 따라 얼마나 불리해지는지 측정하는 것이다.

Vertex와 index 수, mesh와 object 수, geometry buffer 크기, buffer layout, index format, geometry 배치 순서와 screen-space 접근 순서를 변화시킨다.

TVB resolve 시간, 다른 renderer의 geometry 비용, L2 hit rate, VRAM traffic과 cache traffic을 비교해 working set이 cache 범위를 넘을 때의 성능 변화를 모델링한다.

## 실험 6: TVB Material 처리 방식

목적은 TVB에서 material을 분류하고 shading하는 방식에 따라 성능과 scalability가 어떻게 달라지는지 측정하는 것이다.

Monolithic shader, global material list 또는 binning, tiled binning, material별 work queue, material sorting, visibility buffer의 material ID 포함 여부를 비교한다.

Material 수, material의 화면상 공간 분포, material entropy, resolution과 visible pixel 수를 변화시키고 classification 비용, queue 생성 비용, shading locality와 total 시간을 분석한다.

## 결과 분석

각 실험에서는 변수 sweep으로 crossover point를 찾고, 입력 크기로 정규화한 비용과 pass별 회귀를 통해 고정 비용과 선형 비용을 분리한다.

Synthetic scene에서 얻은 모델은 scene fingerprint를 이용해 Intel Jungle Ruins와 Bistro에 적용하며, Donut 측정 결과와 비교해 현실에서도 성립하는지 검증한다.

예측이 맞지 않는 경우 projected triangle size, overdraw, quad efficiency, material locality, geometry layout, cache working set, texture 특성, draw-call 구조와 lighting 부하를 원인 후보로 분석한다.

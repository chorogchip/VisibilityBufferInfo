    기존 Forward / Deferred 렌더링 대비 성능상 유리한 조건을 실험적으로 규명

* 장면 구성과 렌더러 설명에 제대로 된 근거를 붙여 정보 누락 없이 설명할 것.

장면 구성 1: synthetic primitive scene, Custom Renderer
- Crossover Point 찾기: 변수에 따른 Tradeoff를 관찰하고 유리해지는 알고리즘이 바뀌는 지점 찾기. 점을 잘 찍으면 됨.
- 데이터와 맞게 성능 모델을 수립(공식)
- Roofline 분석: VRAM bandwidth, ALU, texture sampling, rasterization, L2 cache 등에 의한 roofline 분석
- 즉, 알고리즘 A와 B는 어떤 변수 v에 따라 유리한 지점이 바뀌고, 그걸 공식으로 이렇게 모델링했고, roofline이 어떻게 된다를 보여주자

장면 구성 2: Intel Jungle Ruins + Bistro, Donut Deferred Renderer
- 실용적인 부하(위에서 부족했던 상수값) 측정
- 인공 씬은 실용적인 부하 분포를 가지지 않으므로 실제 렌더러의 deferred를 baseline으로 삼아 거기에 visbuf를 붙이기.

- Scene fingerprint 추출 (각 씬의 geometry/material/texture/overdraw 특성 수치화)
- 인공 씬에서 세운 모델이 현실에서도 통하는지 가설을 세우고 검증

- buffer format, barycentric/derivative calc, material binning 등 variant 구현

실험 1: VB는 두 번째 래스터화와 quad낭비 비용을 얼마나 줄이는가?
실험 2: VB는 버려지는 프래그먼트들의 texture sampling / material shader 비용을 얼마나 줄이는가?

즉 1번은 앞단, 2번은 중간 단의 비용을 측정하려고 한다.
앞단의 비용은 prepass와 비교시 전체 VS -> raster -> early-Z 까지이다. (forward는 PS와 material까지 비용이 넘어간다)
단 visbuf도 PS에서 VS의 연산을 포함해서 하기에
overdraw의 VS + 전체 raser, early-Z - visutil오버헤드(무게중심) 만큼의 차이가 난다
ALU연산과는 독립적이다

- ALU와는 독립이다
- 전체 geometry에 선형 비례, prepass가 상수 클것이다(raster까지 다하니까 - 이걸 실험)
- overdraw가 많으면 VS의 비용 추가 - 정점 많을수록 상수 클것이다
- overdraw는 물체들의 겹치는 수, 정렬된 정도 등으로 변한다
- VS가 compute bound가 맞을까? 이걸보는건 실험 4에서 한다 (메모리 크기와 roofline 등을 다룸)
- 낭비 quad에 대해서(다 보이는 quad일 경우에만.), prepass는 material까지 넘어가는데 material비용에 따라 갈린다. 즉 이건 2번 실험.
- 안보이는 quad는 상관없지 않을까 예상.
- 낭비 quad는 VS가 아닌 PS에서만 도는거고 material비용과 관련돼있다.

실험 2는 
- forward/deferred는 삼각형이 작을수록 낭비 quad비용이 높고 ALU/texture sampling이 많을수록 비용이 커질것이다
- TVB는 quad 상관없이 상수임
- resolution도 조정가능.

성능 모델을 그리면

forward = (full geometry) (VS cost + raster) + ( (visible geometry) + (overdrawn geometry) + (quad) ) (PS cost)
forward prepass = (full geometry) (VS cost + raster) + (full geometry) (VS cost + raster) + ( (visible geometry) + (quad) ) (PS cost)
즉 forward prepass - forward = (full geometry) (VS cost + raster) - (overdrawn geometry) (PS cost)
다시 말해 overdraw의 PS비용을 줄이기 위해 모든 geometry의 변환과 래스터를 한다

visbuf = (full geometry) (VS cost + raster + visbuf recording) + (visutil) + (visible geometry) (PS cost)
즉 visbuf - forward prepass = (overdraw) (visbuf recording) + (visutil) - (full geometry) (VS cost + raster) - (quad) (PS cost)
다시 말해 prepass비용을 생략하고 (모든 geometry를 다시 변환 생략하고) quad의 PS 비용을 줄이기 위해 full geometry의 visbuf recording과 상수 visutil을 한다

그러면 visbuf recording은 사실 overdraw를 제외하면 씬에 따라 크게 변할 게 없고 (overdraw가 상대적으로 영향을 더 주겠군! 이건 이미 실험결과로 나옴)
VS비용이 증가할수록 이득, quad 낭비와 ALU가 증가할수록 이득이겠군!

virbuf - forward = (overdraw) (visbuf recording) + (visutil) - (overdrawn + quad geometry) (PS cost) 이므로
forward에 비해선 똑같이 full recording을 지불해 overdraw와 quad의 PS비용을 아끼겠군
- 그럼 ALU비용이 낮으면 어쩌면 forward의 민감도보다 visbuf의 민감도가 클지도 모르는걸? 이아니라 얘는 32비트 두개여서 그런 일이 안 일어날듯 일단 실험은 해보자
- 이 비교는 overdraw 상수를 비교할 떄 좋을 거 같고, 또 overdraw가 없을 때
- (visible + quad) (visbuf record) + (visutil) - (quad) (PS) 이므로 visible/visbuf record는 overdraw없을떄 상수이므로 quad의 차이가 나겠군 그 차는 ALU비용이 결정

- 그러면 ALU비용에 따라서, overdraw비용에 따라서, forward와 visbuf가 경쟁할 수 있겠고 (둘다 낮을떄만 forward가 유리하겠고)
- geometry의 양 자체는 overdraw가 같다면 visbuf에 불리할 수 있다(미소하게)
- overdraw가 더 심해질 때 geometry가 많다면 ALU가 적을 때 visbuf가 오히려 불리할수도

forward + prepass 와 비교하면
- overdraw가 많을수록 visbuf가 불리하겠고
- full geometry의 cost를 prepass가 항상 부담하므로 geometry가 늘면 visbuf가 유리하겠네
- quad가 늘면 visbuf가 유리하겠고

변수는 G, O, Q, A 인데
컨트롤가능한건 A, 장수, 순서, division이었다.
G = 장수 * division^2
O = f2(순서, 장수)
Q = f3(division)
A = ALU
물론 선형으로 곱해지거나 더해지는 상수도 포함

일단 Q의 경우..  음
일단 순서의 경우 앞에 역순 부분 - 뒤에 정순 부분 하면은
앞의 역순 부분만큼만 overdraw가 됨.
즉 이걸로 선형으로 컨트롤 가능.

Q는 대략 사각형을 그릴때 O(n^2)개의 픽셀에서 O(n)개의 quad낭비가 발생하므로 1/n의 선형변환만큼임.
따라서

G = 장수 * division^2
O = 순서, 순서 < 장수.
Q = 1/(width/division) = division
A = ALU

O가 골치아프므로 장수는 기본적으로 많이 확보.
순서는 overdraw가 30만 돼도 꽤 힘드므로 최소 장수가 30으로 하면 문제없음.

그리고 grey box 접근식을 사용
일단 변수는 15개

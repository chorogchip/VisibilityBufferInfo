목적: Triangle Visibility Buffer(TVB) 기법이
    기존 Forward / Deferred 렌더링 대비 성능상 유리한 조건을 실험적으로 규명


장면 구성 1: synthetic primitive scene
- Crossover Point 찾기: 변수에 따른 Tradeoff를 관찰하고 유리해지는 알고리즘이 바뀌는 지점 찾기. 점을 잘 찍으면 됨.
- 데이터와 맞게 성능 모델을 수립(공식)
- Roofline 분석: VRAM bandwidth, ALU, texture sampling, rasterization, L2 cache 등에 의한 roofline 분석
- 즉, 알고리즘 A와 B는 어떤 변수 v에 따라 유리한 지점이 바뀌고, 그걸 공식으로 이렇게 모델링했고, roofline이 어떻게 된다를 보여주자

장면 구성 2: Intel Jungle Ruins + Bistro, Donut Deferred Renderer
- 실용적인 부하(위에서 부족했던 상수값) 측정
- 인공 씬은 실용적인 부하 분포를 가지지 않으므로 실제 렌더러의 deferred를 baseline으로 삼아 거기에 visbuf를 붙이기.

- Scene fingerprint 추출 (각 씬의 geometry/material/texture/overdraw 특성 수치화)
- 구에서 세운 모델이 현실에서도 통하는지 가설을 세우고 검증
- HW 프로파일링을 동반
- 안 통한다면 이유를 찾기
- TVB 약점, 강점 케이스 찾기
- material binning / sorting 실험 (확장)

실험 1-1 : geometry처리 절약 - overdraw 실험
- material=1, mesh=1, 인스턴스를 많이 사용합니다.
- 큰 삼각형으로 된 화면을 가득 채우는 직사각형을 여러 개 배치해 overdraw를 조절합니다.
- forward pre-pass에 집중해 비교합니다.
- 물체의 정렬 순서에 따라(앞부터, 뒤부터, 무작위) 비교합니다.
- scalability를 비교하고 성능 모델을 세우는 것이 목표입니다.

- overdraw를 많이 했음에도 예상과 달리 forward + prepass의 depth prepass와 forward pass의 비용이 물체 정렬 순서에 상관없이 거의 같게 나왔습니다.
	- 셰이더 비용이 낮다고 생각해 ALU연산을 넣어 셰이더 비용을 증가시킨 후 실험했지만 결과는 비슷했습니다.
	- 한 draw call 내부의 인스턴스들의 정렬 순서는 영향이 적을 수 있다는 것을 발젼해 여러 draw call 로 쪼갠 후 정렬 순서를 달리 해 실험했습니다.
	- 최종적으로 오브젝트들을 정렬하는 코드에서 오류가 있었던 것을 확인했고, 올바르게 작성하여 테스트한 결과 정렬 순서와 알고리즘에 의존해 성능이 변하는 결과가 나왔습니다.
	- 조금 더 실험을 이어나가 잘 설명할 수 있게 다듬으려고 합니다.

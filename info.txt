1. 전통적인 Direct Draw
for object:
    BindPipeline()
    BindResources()
    DrawIndexed()
장점

가장 구현하기 쉽고 디버깅하기 쉽습니다.

mesh/material별 PSO 전환이 자연스럽다.
alpha test, two-sided, skinning, WPO 같은 특수 케이스를 분리하기 쉽다.
RenderDoc이나 PIX에서 draw 단위로 추적하기 쉽다.
Visibility Buffer의 attribute reconstruction에 집중할 수 있다.

그리고 포트폴리오에서는 direct path가 쓸모없는 코드가 아닙니다. 이후 indirect, mesh shader 경로의 정확성 및 성능 기준점이 됩니다.

단점

오브젝트나 meshlet마다 draw call을 생성하면 CPU submission 비용과 상태 관리 비용이 증가합니다. 특히 CPU가 매 프레임 visibility를 결정하고 command buffer까지 다시 생성하면 대규모 장면으로 확장하기 어렵습니다.

어떤 포트폴리오에 적합한가

Visibility Buffer 자체를 보여주려는 경우입니다.

barycentric reconstruction
normal/UV 복원
texture gradient
G-Buffer와의 비교
material shading 분리

이쪽이 핵심이라면 굳이 처음부터 indirect를 넣을 필요는 없습니다.

평가
구현 난이도: 3/10
포트폴리오 차별화: 4/10
기준 경로로서의 가치: 매우 높음

반드시 남겨두되 최종 경로로만 제시하지 않는 것이 좋습니다.

3. GPU-Generated Indirect Draw

    ↓

    ↓
Command Compaction
    ↓
Indirect Argument Buffer
    ↓
DrawIndexedIndirectCount / ExecuteIndirect

가장 현실적인 modern renderer 선택입니다.

Khronos의 공식 Vulkan 샘플도 GPU에서 frustum culling과 indirect command 생성을 수행해 CPU 사용량을 줄이고, 여러 draw를 한 indirect 호출로 실행하는 구조를 보여줍니다.

장점
CPU가 개별 오브젝트를 제출하지 않는다.
instance 수가 증가해도 CPU frame time이 비교적 안정적이다.
HZB occlusion culling과 연결하기 좋다.
Visibility Buffer와 궁합이 좋다.
bindless geometry/material 구조를 자연스럽게 요구한다.
필요한 구조

GPU-generated indirect를 제대로 하려면 렌더러 데이터 구조도 바뀌어야 합니다.

SceneInstanceBuffer
MeshMetadataBuffer
MaterialInstanceBuffer
GlobalVertexBuffer
GlobalIndexBuffer

Visibility Buffer에는 단순한 SV_PrimitiveID만 기록하면 부족할 가능성이 큽니다. primitive ID는 draw 내부에서만 의미가 있을 수 있으므로 보통 다음 중 하나가 필요합니다.

(drawID, primitiveID)
(instanceID, meshID, primitiveID)
(meshletID, localPrimitiveID)
(globalPrimitiveID)

D3D12에서는 indirect command마다 root constant나 resource 주소 등을 전달할 수 있고, 나머지 pipeline state는 command list로부터 상속됩니다.

가장 중요한 제약: 임의의 머티리얼 PSO 전환

일반적인 multi-draw indirect 호출 내부에서는 draw마다 임의의 graphics pipeline을 바꾸기 어렵습니다.

즉 이런 구성은 곤란합니다.

Indirect Draw 1 → Masked PSO
Indirect Draw 2 → WPO PSO

Bind Masked PSO

Bind WPO PSO

포트폴리오 가치: 매우 높음

현실적인 포트폴리오에서는 가장 추천하는 제출 방식입니다.

4. Device-Generated Commands와 Work Graphs

Indirect draw보다 더 나아가 GPU가 shader나 pipeline 선택까지 포함한 작업 그래프를 생성하는 방향입니다.

Vulkan의 Device Generated Commands 제안도 기존 indirect rendering이 pipeline별 command buffer 분리나 불필요한 메모리 round trip을 요구할 수 있다는 문제를 다룹니다.

장점
GPU가 작업 생성부터 shader 선택까지 더 많이 담당할 수 있다.
매우 많은 material/shader permutation을 다루기 좋다.
scene traversal, LOD, culling, dispatch generation을 완전히 device-driven으로 구성할 수 있다.
단점
API와 하드웨어 의존성이 강하다.
디버깅이 어렵다.
Visibility Buffer 포트폴리오의 주제가 흐려진다.
구현량에 비해 시각적으로 설명하기 어렵다.
평가
구현 난이도: 9/10
포트폴리오 위험성: 높음
연구적 가치: 높음

첫 프로젝트에서는 제외하는 편이 낫습니다.

8. Mesh Shader를 쓸 것인가

Mesh shader는 indirect draw나 software rasterizer를 대체하는 기능이 아닙니다.

Mesh shader는 기존 VS/HS/DS/GS 중심 geometry frontend를 대체하고, threadgroup이 vertex와 primitive 묶음을 직접 생성합니다. meshlet 단위 culling이나 primitive culling을 rasterizer 앞에서 수행할 수 있으며, DispatchMesh도 direct 또는 indirect로 실행할 수 있습니다.

즉 다음 조합이 모두 가능합니다.

Direct DispatchMesh + HW Raster
Indirect DispatchMesh + HW Raster
Mesh Shader + Visibility Buffer
Mesh Shader output을 SW raster 입력으로 사용
장점
meshlet 단위 처리에 자연스럽다.
vertex 공유를 threadgroup shared memory로 제어할 수 있다.
meshlet 전체를 output 전에 제거할 수 있다.
cone culling, frustum culling, primitive culling과 잘 맞는다.
per-primitive attribute로 visibility metadata를 넘기기 편하다.
비용

mesh shader 하나만 작성하는 것이 아니라 content pipeline도 바뀝니다.

meshlet 생성기
meshlet vertex remap
local triangle index
meshlet bounds
normal cone
데이터 압축
runtime decoding
non-mesh-shader fallback

Microsoft의 사양도 mesh shader를 사용하려면 geometry를 meshlet 같은 압축된 batch로 재구성하고, 기존 input assembler 경로와 다른 toolchain이 필요하다고 설명합니다.

Amplification Shader까지 필요한가

처음에는 필요 없습니다.

더 현실적인 구조는 다음입니다.

Compute Shader:
    instance/meshlet culling
    visible meshlet list 생성

Indirect DispatchMesh:
    visible meshlet만 mesh shader 처리

Amplification shader까지 넣으면 nested work generation과 payload 설계가 추가됩니다. 기술적으로 멋있지만 포트폴리오 완성에는 도움이 적을 수 있습니다.

가장 좋은 포트폴리오 구성

동일한 meshlet 데이터에 두 개의 경로를 제공합니다.

Path A:
Compute Culling
→ Indexed Indirect Draw
→ Vertex Shader
→ HW Raster

Path B:
Compute Culling
→ Indirect DispatchMesh
→ Mesh Shader
→ HW Raster

그러면 “mesh shader를 썼다”가 아니라 다음을 비교할 수 있습니다.

meshlet당 처리 시간
primitive rejection
vertex load 수
geometry pass 시간
triangle 크기에 따른 성능
meshlet 크기별 성능
평가
구현 난이도: 7~8/10
실무 관련성: 높음
SW raster보다 완성 가능성: 훨씬 높음

고급 기능 하나를 선택한다면 hybrid raster보다 mesh shader가 현실적입니다.

장점
visibility pass shader permutation이 거의 없다.
모든 opaque geometry를 큰 indirect batch로 묶기 쉽다.
visibility buffer bit 수를 아낄 수 있다.
가장 순수한 Visibility Buffer 구조다.

원래 Visibility Buffer 논문도 triangle ID와 instance ID를 기록하고, 후속 단계에서 visible triangle data와 surface material을 찾아 shading합니다.

적용 가능한 대상
static opaque
단순 skinned mesh
material이 vertex position이나 coverage를 바꾸지 않는 경우
9-2. Raster Behavior별 Material Pass

일부 material은 visibility 자체에 영향을 줍니다.

alpha mask
world position offset
displacement
pixel depth offset
two-sided
custom culling
procedural geometry

이것들은 default opaque pass와 완전히 동일하게 처리할 수 없습니다.

따라서 material instance별 패스가 아니라 raster behavior category별 패스를 만드는 것이 좋습니다.

RasterCategory 0: Default Opaque
RasterCategory 1: Two-Sided
RasterCategory 2: Alpha Masked
RasterCategory 3: WPO
RasterCategory 4: Skinned

Epic도 Nanite hardware rasterizer를 material shader 형태로 바꾸면서 WPO, pixel depth offset, masked material 같은 programmable raster 기능을 확장한 바 있습니다.

핵심 원칙

1,000개의 material instance가 있더라도 raster PSO는 1,000개가 아니라 소수여야 합니다.

Material Instance:
    texture indices
    scalar parameters
    colors

Material Shader Category:
    actual shader code / raster behavior

방식 A. Fullscreen Uber Shader
for every pixel:
    materialID = ResolveMaterial(visibility)

        case StandardPBR:
        case ClearCoat:
        case Foliage:
        ...
장점
가장 구현하기 쉽다.
classification pass가 필요 없다.
한 번의 fullscreen compute dispatch로 끝난다.
단점
wave 내 material divergence
큰 shader binary
많은 resource 접근
shader permutation 관리가 어려워진다.

추천 용도

첫 번째 working implementation.

처음에는 이 방식으로 correctness를 완성한 뒤 classification을 추가하는 것이 좋습니다.

방식 B. 머티리얼마다 Fullscreen Pass
for each material:
    fullscreen dispatch
    if pixel.material != currentMaterial:
        return
장점
구현이 단순하다.
각 material shader를 독립적으로 작성할 수 있다.
단점

material 수에 따라 화면 전체를 반복해서 스캔합니다.

Cost ≈ ScreenPixels × MaterialCount

material이 두세 개뿐인 데모가 아니라면 좋지 않습니다.

포트폴리오 최종 구조로는 추천하지 않습니다.

방식 C. Material Tile Classification
Visibility Buffer
    ↓
8×8 또는 16×8 Tile Classification
    ↓
Material Category별 Tile Worklist
    ↓
Material별 Compute Dispatch

예를 들어 한 타일에 Standard PBR과 Clear Coat가 보인다면 해당 타일을 두 worklist에 넣습니다.

StandardPBRTiles = [0, 1, 5, 8, ...]
ClearCoatTiles   = [5, 12, ...]

각 material shader는 자신이 필요한 tile만 실행합니다.

원래 Visibility Buffer 연구도 surface material마다 shading이 필요한 tile worklist를 만들고, material별 shading kernel을 dispatch하는 방식을 사용했습니다.

장점
material shader를 분리할 수 있다.
fullscreen 반복을 피한다.
픽셀 spatial locality가 유지된다.
texture gradient와 neighborhood 연산에 상대적으로 유리하다.
구현 난이도와 성능의 균형이 좋다.
단점

타일 안에 해당 material 픽셀이 하나뿐이어도 전체 tile thread가 실행될 수 있습니다.

material이 잘게 섞인 장면에서는 여러 material worklist에 동일 타일이 반복해서 들어갑니다. 그래서 Material Complexity per Tile을 시각화하는 것이 좋습니다. Epic의 Nanite 관련 도구도 타일 내 unique material 수를 material coherence 지표로 사용합니다.

포트폴리오 추천도

가장 높음

특히 Visibility Buffer의 texture gradient 문제가 더 까다로워집니다. adjacent lane이 adjacent pixel이라는 보장이 없기 때문입니다.

이 구성에서 기본 구현은 indirect + traditional VS + HW raster로 잡는 것이 좋습니다.

Mesh shader는 대체 경로로 추가하고, software raster는 정말 시간이 남을 때 별도의 micro-triangle 실험으로 제한하는 게 맞습니다.

가장 중요한 판단은 이것입니다.

포트폴리오에서 Nanite와 비슷해 보이는 시스템을 만드는 것이 목표가 아니라, 각 경로의 비용과 이득을 분리해서 측정할 수 있는 렌더러를 만드는 것이 목표여야 합니다.

따라서 우선순위는 다음이 좋습니다.

1. HW Visibility Buffer correctness
2. GPU-generated indirect
3. Material tile classification
4. Meshlet culling
5. Mesh shader A/B path
6. HW/SW hybrid raster

Indirect + HW raster + tile-classified material shading까지 완성하면 이미 충분히 강한 포트폴리오입니다. Hybrid software raster는 그 프로젝트를 한 단계 더 깊은 연구물로 바꾸지만, 완성 가능성을 크게 떨어뜨리는 선택입니다.
